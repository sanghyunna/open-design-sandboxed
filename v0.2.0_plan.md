# v0.2.0 Plan — Media subsystem removal, Live Artifacts excision, Report chip, deck fixes

> Status: **Plan / not started.** Research complete (9 read-only Opus agents + first-hand
> verification). No code has been changed. This document is the execution contract for v0.2.0.

---

## Context — why this release exists

This is a focused corporate fork of Open Design (codex + cursor only; AMR / social / telemetry
already stripped in prior waves). v0.2.0 continues that narrowing: the product is being
re-centered on **document / deck / prototype generation driven by coding agents**, and the
media-generation surface (image / video / audio / HyperFrames) and the Live Artifacts runtime
are being removed wholesale because they are not part of this fork's use case. Alongside the
removals we add a **Report** output type (a flowing long-form document, a sibling to the slide
deck) and fix two long-standing slide-deck UX bugs (double arrow-advance, single-slide PDF
export).

The intended end state: a leaner Home Hero (no media chips), a smaller daemon (no media
generation, no live-artifacts runtime), a simpler Settings dialog (no Media providers editor),
intact core chat / BYOK / project-creation / memory / research / X-search, plus a working
Report chip and reliable deck navigation + PDF export.

## Global execution principles

1. **Red spec first for the two bugs (tasks 8, 9).** Repo convention: write a falsifiable test
   that goes red on `main` before the fix, green on the branch. Homes already identified
   (below). For the removals, lean on `pnpm typecheck` + updated unit/e2e suites as the safety
   net.
2. **Home Hero edits are per-chip and localized.** Tasks 1/2/3/4/5 (remove 5 chips) and task 6
   (add 1 chip) each touch `chips.ts` + the two duplicated label switches (`HomeHero.tsx`,
   `HomeView.tsx`) + i18n (en/ko). These are small, isolated array-entry / switch-case edits, so
   under the 4-PR split **each PR edits only its own chip(s)** and **re-verifies the current state
   of those four shared files at the start of the PR** to avoid drift. (If you instead land
   everything on one branch, collapse them into a single coordinated pass — but the split is the
   chosen path, so per-PR is the default.)
3. **Dual-surface rule.** Every capability change touches web UI **and** the `od` CLI. For
   removals: also remove the matching CLI surface (`od media …`, `od mcp/tools live-artifacts`).
   For Report: verify project-creation parity via existing `od project` / plugin-apply (no new
   subcommand expected).
4. **Surgical vs delete.** `media.ts` is a single dispatcher — but with all surfaces removed it
   can be deleted wholesale. `mcp-live-artifacts-server.ts` and `tool-tokens.ts` are **shared
   with connectors** — these must be SPLIT, never deleted.
5. **Full type purge (locked).** Remove `image|video|audio` from `ProjectKind` / `MediaSurface`
   / analytics enums and fix every exhaustive `switch` / `Record<>` across web + daemon +
   contracts. `pnpm typecheck` is the driver — it will enumerate every site.
6. **No co-author trailers** in commits (repo policy). Branch off `main`.

### WS-A — Deck arrow double-advance (task 8) · complexity **S**

**Root cause (confirmed):** the framework deck's `onKey` is a capture-phase listener bound to
**both** `window` and `document` with no per-event dedupe guard, so one physical ArrowRight in
the focused iframe invokes it twice → `idx += 2`. Sibling decks (`simple-deck`,
`taste-editorial`) bind the same two targets but are single-step **because** they carry a guard
the framework omits.

**Fix:** add the sibling guard `if (e.defaultPrevented) return;` as the first line of `onKey`
(mirrors `design-templates/simple-deck/example.html:146`) **or** bind only `window` (drop the
`document` capture line). Apply to all **three** synced copies and update the prompt directive
that currently tells models to bind "both targets":
- `apps/daemon/src/prompts/deck-framework.ts` (`onKey` 290–297, binds 301–302, directive ~367)
- `packages/contracts/src/prompts/deck-framework.ts` (identical mirror 290–302)
- `templates/deck-framework.html` (identical copy 226–238)

**Red spec:** `apps/web/tests/runtime/design-template-deck-nav.test.ts`. Import
`DECK_SKELETON_HTML`, load with `runScripts:'dangerously'`, dispatch **one** `keydown ArrowRight`
at `document.body` (must target a node inside the document so it traverses window→document
capture), assert active slide index === **1**. Red today (lands on 2), green after fix.

### WS-B — PDF prints one slide (task 9) · complexity **S**, risk **M**

**Root cause (confirmed):** the injected `DECK_PRINT_CSS` safety-net sets size + `page-break`
but **never overrides `display`/`visibility`/`opacity`**. Class-toggle decks hide inactive
slides with `.slide:not(.active){display:none !important}`; when a deck's own `@media print`
block is missing or was edited away, inactive slides stay `display:none` through print → only
the active slide renders.

**Fix:** harden the slide rule in `DECK_PRINT_CSS` to force-reveal every slide —
`display:flex !important; visibility:visible !important; opacity:1 !important; inset:auto
!important` — and broaden the selector list. This mirrors what the canonical framework's own
print CSS already does (`deck-framework.ts:197`), so it's consistent, not a new regression.
Apply to **both** copies:
- `apps/web/src/runtime/exports.ts` (`DECK_PRINT_CSS` 1073–1106, `injectDeckPrintStylesheet` 1108)
- `apps/desktop/src/main/pdf-export.ts` (duplicate 22–55, `buildPrintableDocument` 244–248)

Add a **copy-identity guard test** asserting the two `DECK_PRINT_CSS` blocks stay byte-identical
(cannot share a module — desktop can't import web `src`, and `packages/contracts` is
web/daemon-only). Put it where both copies are reachable from one test runner —
`apps/web/tests/runtime/exports.test.ts` reading the desktop copy via a relative `fs.readFile`,
or a small `e2e/tests/` consistency check (e2e is allowed to observe >1 app boundary).

**Red spec (assert behavior, not just the CSS string):** in
`apps/web/tests/runtime/exports.test.ts` (existing deck-PDF case at 477), add a case:
`exportAsPdf` with a class-toggle deck containing `<style>.slide:not(.active){display:none}</style>`,
**two** `.slide` sections, no own `@media print`, `{deck:true}`. Assert the produced print
document makes **both** slides print-visible — i.e. the injected rule resolves inactive `.slide`
to a visible `display`/`visibility` (not merely that a `page-break` string is present). Red today.
Mirror against `buildPrintableDocument` in
`apps/desktop/tests/main/save-print-ready-document-as-pdf.test.ts` so the desktop copy is covered
too. Pair with a manual regression on an **intact** `deck-framework`/`html-ppt` deck to confirm
force-reveal does not distort a deck that already prints correctly (the one behavioral risk a
string test can't see).

> **Risk note:** forcing `display:flex` flattens grid/block active-slide print variants — but the
> framework already does this, so parity is acceptable. Regression-check decks that already print
> correctly (intact `deck-framework`, `html-ppt` family) so the force-reveal doesn't distort them.
