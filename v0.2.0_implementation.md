## 2. How it was built — orchestration

Per your directive, the work was driven by **three Opus "middle-manager" subagents** (one per task-group), each **orchestrating only** and delegating the actual edits/review to **Codex `gpt-5.5` `xhigh`** agents over the Codex `app-server` JSON-RPC protocol, with an **Opus alignment reviewer** gating each group's "DONE".

- **Harness built + smoke-tested:** a write-mode Codex driver (`codex_agent.mjs`) — spawns a fresh `codex app-server`, runs one autonomous `gpt-5.5`/`xhigh` turn in `danger-full-access` sandbox, auto-approves exec/patch requests; plus a read-only review mode returning a strict JSON verdict. Protocol/params derived from the generated TS bindings.
- **Group split & concurrency:** **G3 (deck fixes)** and **G1 (media teardown, XL)** ran **in parallel** — G3 on the main tree, G1 in an isolated `git worktree` (disjoint file sets) — then G1 was merged into `v0.2.0` conflict-free. **G2 (Live Artifacts + Report)** ran on the merged tree (it shares `chips.ts`/contracts with G1, so it was sequenced after).
- **Restart resilience:** the environment restarted the process mid-run twice; managers were switched to **commit after every verified sub-task**, so progress survived.
- **Codex reviewer drill:** fresh Codex `app-server` instances were used to review diffs (structured JSON verdict), separate from the implementing agents.
- **Pivot to parallel Sonnets:** G2's Opus manager repeatedly stalled at coordination points (the environment auto-backgrounds long Codex turns, so the manager kept yielding). Per your instruction ("run multiple sonnets, not a lazy Opus"), the G2 finish was taken over directly with **fleets of parallel Sonnet workers editing files directly** (dropping the Codex indirection for speed/reliability): e.g. a 6-way file-disjoint fan-out for the live-artifacts removal, a dedicated worker for the 1,166-line surgical removal from `FileViewer.tsx` (precise edits only, to avoid an encoding-corruption that a full rewrite had caused), and a 4-way fan-out for the Orbit excision.
- **Verification-driven:** `pnpm typecheck` was the completeness driver (esp. the media type-purge and the Orbit cascade); a **typecheck-batch loop** (run typecheck → feed bounded error batches to workers) reconciled cross-cutting removals.

Alignment gates: **G3 → "perfect alignment, without caveats"** ✓ · **G1 → "perfect alignment, without caveats"** ✓ · **G2 → "perfect alignment, without caveats"** ✓ (reached after the Orbit-content remnant fix in §7).

A later session (see §8) added a fourth layer on top of the alignment gate: an independent 11-dimension audit that treated every prior "done"/"aligned" claim — including this document's — as unverified, and re-derived every fact from the current working tree with real tool calls (Read/Grep/Bash), plus a live manual verification pass against the actually-running app. This caught real gaps the alignment gate had missed.

### 8.4 Real packaging bug found only by attempting an actual build

None of typecheck/guard/unit-or-component-tests/the audit above cover `tools/pack` (the Electron packaging tool). Attempting an actual `build-portable.ps1` run surfaced a real crash: the packager's resource-tree phase does an `lstat` on a hardcoded top-level `prompt-templates/` directory (`tools/pack/src/resources.ts`'s `BUNDLED_RESOURCE_TREES` list + a cache-key hash in `tools/pack/src/win/resources.ts`) — a directory this release deletes. `ENOENT` on every build attempt.

Fixed: removed the dead `{ from: "prompt-templates", to: "prompt-templates" }` bundling entry, removed the corresponding `promptTemplates: await hashPath(...)` cache-key line, bumped `RESOURCE_TREE_CACHE_SCHEMA_VERSION` 8→9 (invalidates stale caches), and updated the 3 test files that had built fixtures/assertions around bundling that directory (`tools/pack/tests/{resources,win-resources,mac}.test.ts`). Verified: 32/32 `tools-pack` tests pass, `tools-pack` rebuilds with zero remaining references, `pnpm guard` clean.

Separately, the default `build-portable.ps1` invocation on this machine hit a reproducible Windows `EPERM` on a cache-directory rename during the workspace-build phase (a transient file-lock, likely AV/indexer scanning — this repo's own `AGENTS.md` documents Windows-native packaging as "best-effort" with a history of exactly this class of friction). Worked around by invoking the script with `-PortableZipCompression 6`, which reroutes the build cache to a fresh subdirectory (an existing, supported script parameter — no script changes needed). Portable build succeeded: `Open Design-rg-portable.zip`, 263.4 MB, dropped at `D:\dev\open_design_port\` per the project's zip-drop convention.
